Beslissing 8s5 — "Slim antwoord"-kaart op /zoeken (ZOEK-3 · #2266)

Aanbeveling: B ★  Getinte spotlight. Het AI-answer is precies de feature die ZOEK-3 wil "surfacen" — dus moet het zich visueel onderscheiden van de lexicale rijen (anders had A net zo goed gekund). B doet dat eerlijk: duidelijk gelabeld als AI, bronnen als doorklikbare links (de mitigatie die 7o8 bij C miste), en bínnen het bestaande palet — geen nieuwe merkkleur. A = veiligste/laziest (maar het antwoord verdwijnt in de ruis); C = karaktervolst (maar speels rond LLM-prose is een toon-risico). Kies A / B / C.

  • Score-gate (alle varianten): top ≥ 0.5 → prose-kaart; 0.35–0.5 → "Gerelateerd" (rijen, geen prose); < 0.35 → niets (enkel lexicaal). Backend kent score + answer al.
  • Bron-URL's: article → /nieuws/{slug} · page → /club/{slug} · responsibility → /hulp#{slug} (HubSearch-hashconventie).
  • Augment, geen vervanging: lexicaal (spelers/teams/staf/nieuws + filters) blijft ongewijzigd eronder. Semantisch indexeert géén spelers/teams.
  • Stille terugval: endpoint-fout / 503 → blok rendert niet, lexicaal onaangetast. Geen "AI"-hint, geen error.
  • Na keuze: TDD-first component (SearchAnswerCard) + Storybook-story (autodocs) + check-all, zoals 8s4.
