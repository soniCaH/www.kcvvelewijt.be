Phase 7 · /club/organigram · 7o2b — THE FINDING MODEL

Resolving the real question: how do the search box, the Hulp finder, and the browse sections work together for someone looking for something specific? First — what the search actually returns (it spans people AND answers). Then three coherent ways to organise the page around it. This decides the finding model before we lock 7o2.

First: what one search box returns.

The existing search ranks across people (name · function) and hulpvragen (question · keywords · summary). Two query kinds, two result types — typed and clearly labelled, so a click goes to the right place.

Search is the verb. The sections are the nouns.

One box up top answers both intents (a person or an answer). The three sections below are the same dataset reached by browsing instead of typing — Structuur = browse people, Verkenner = browse relationships, Hulp = browse the answers by topic. Nothing is duplicated: Hulp isn’t a second search, it’s the catalogue you land in if you’d rather click categories than type. The fast user types; the browser scrolls.

Best for: clearest “how it fits”, keeps the loved structure-explorer, one mental model.

Cost: the Hulp section must read as “browse the answers”, never as another search box.

Pick your door first.

Two explicitly separate lanes. A people-search feeds the structure half; a question-finder is its own half. Each search is scoped to one domain, so results never mix. The page literally splits into “De mensen” and “De hulplijn”.

Best for: zero overlap, each box does exactly one thing.

Cost: two search boxes; the user must self-diagnose which intent they have before typing.

Answer first, org chart second.

Inverts the emphasis: the page leads with the problem-solver (because most end-users come with a question, not a name). The org structure + verkenner become reference material lower down. The hero box is a question-finder that also surfaces the matching contact.

Best for: a help-desk-first club site; demotes the org chart.

Cost: buries the structure explorer you specifically loved; weaker as an “organigram” page.

Decision 7o2b — the finding model

I ★ one search (people + answers) as the fast path, the three sections as the browse-the-same-data backup — clearest “how it all fits”, keeps the structure explorer central. II two scoped boxes / two halves — zero overlap but two doors. III Hulp-led — answer-first, structure demoted.

Orthogonal call (any model): search intelligence. Today = keyword/substring ranking (free, exists). Upgrade = semantic/AI over the responsibility docs (the bge-m3 + Vectorize stack already powering /zoeken) so natural language like “mijn kind heeft zich bezeerd” finds the medical contact without a keyword match. Real value for the Hulp job, but it’s extra BFF/embeddings build — a scoped issue, not free.
