6.B.d3 · Team identification on event rows

The locked spec puts a mono caps short code on the right of every row (option A below). Owner feedback during implementation: "it's not clear at all which team a player/event is for." Five alternatives drawn from the existing primitives + locked vocabulary. Each is shown with the same fixture (8 events; KCVV at home).

A — Mono caps short code (currently locked)

No change · zero diff

Pro: matches the lock-doc spec verbatim; cheapest; consistent with `` vocabulary. Con: at small size + low contrast, the short codes (KCVV / MECH) read as decorative metadata. New readers don't connect "MECH" to "RC Mechelen" instantly.

B — Logo only (20px circle)

Mirrors `` shield treatment

Pro: universally legible — readers recognise the club at a glance. Same shield asset already loaded by ``. Con: drops the mono caps element from the row, so rows feel "softer" / less editorial. Heavy reliance on PSD shield quality (variable). Initial-letter fallback when no logo.

Belt-and-braces · best signal per row

D — Left/right alignment per side

Familiar pattern (BBC Sport / FotMob)

Pro: spatial signal is the strongest possible — home left, away right, no decoding required. Glyph moves to the team-specific side too. Con: halves the available player-name column on each side; long names truncate harder. Wider gap in the middle feels "scoreboard" rather than "matchverslag". Two extra columns add mobile-collapse complexity.
