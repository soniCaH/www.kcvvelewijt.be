Phase 6.B · Drill round 1 · `<MatchStatusBadge>`

How should the status corner stamp look?

Four status values render a badge: finished (FT), forfeited (FF), postponed (PP), stopped (STOP). scheduled renders no badge (the kickoff time in the hero IS the status).

Three visual treatments compared. Each variant shows the four badges (a) in isolation as a horizontal row + (b) in-context as a corner stamp on a tiny hero card so the size + visual weight read realistically.

Pick one treatment · all 4 statuses ship with it
Treatment 1

Current pill-jersey.

As shipped today: <MonoLabel variant="pill-jersey"> — solid bright-green pill, ink text. Used for postponed + stopped + forfeited; finished is unimplemented (returns null). Bright --color-jersey is the colour the redesign otherwise retired in favour of jersey-deep.

Treatment 3

Direction D + status tint.

Same Direction D chrome as T2 (border-2 ink + shadow-paper-sm + mono caps), but the background tints differ per status to signal severity: FT = cream (neutral), FF = warm (notice), PP = cream-deep (muted, on hold), STOP = card-red (alert).

Notes

  • Reader sees severity at a glance — FF / PP / STOP stand out from the routine FT

  • Tints use existing tokens: warm (already shipped for kicker variants), cream-deep (existing neutral fallback), card-red (would need a token — currently only used in test status colouring)
  • Risk: adding red to the redesign palette where it wasn't before. --color-card-red doesn't exist as a token today; STOP is so rare that a token decision for one badge may be over-engineered
  • Could downgrade STOP to use the warm tint (shared with FF) — STOP is rare enough that distinguishing it from FF visually may not matter
