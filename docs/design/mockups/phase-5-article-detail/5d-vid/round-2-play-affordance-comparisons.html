videoBlock — play affordance (native / jersey-deep disc / corner pill)

Drill 4.2. Locks how the upload path signals "click to play" before the video starts. Locked Round 1 frame (flat TapedFigure + single ochre tape) is held constant across all three options.

Only applies to the upload path. The embed path (YouTube/Vimeo) always renders the provider's own UI inside a privacy-enhanced iframe — nothing we draw influences that. Behaviour: on click, the custom button hides and the native HTML5 controls take over for the rest of the playback (B and C).

Caption typography is held constant (italic Freight body-sm — same as the locked articleImage figcap). Three options for the play affordance only.

Option A · Native HTML5 controls
No KCVV-branded button — browser draws its own
<video controls poster=…> with no overlay. Browser draws a translucent centered triangle on the poster and renders its full controls bar on play. Zero custom code, zero KCVV branding inside the player. Visually inconsistent with the cream surface — controls are dark/translucent.
Vocabulary ✓ zero custom UI
✗ browser chrome doesn't match cream surface
✓ accessible by default (browser handles focus/keys)

Behaviour
  • B and C both render the custom button + poster before play. On click, the button + poster hide and the native HTML5 controls take over for scrubbing, fullscreen, volume, etc. — no custom controls bar.
  • A skips the custom step entirely; the browser handles everything from first paint.
Accessibility
  • B and C need aria-label="Speel video af" on the button. Focus styles follow the design-system button family (existing ring on focus-visible).
  • A inherits browser-default accessibility. Screen reader announces "media player" with native controls.
Out of scope
  • Embed path (YouTube/Vimeo) — provider UI always; nothing to decide here.
  • Width behavior — drilled separately in Round 3 (likely inherits articleImage's width enum).
