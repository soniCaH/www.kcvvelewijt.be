Open 0ms (instant)
Close 0ms (instant)
Easing none
Reduced motion already complies (no motion to reduce)

Reads as: mechanical chrome. Snappy and decisive — the panel either is there or isn't. No artistic flourish. Closest to the legacy `Navigation.tsx` behaviour and to native OS menus. Doesn't add any "fanzine" character but doesn't fight against it either — it stays out of the way.

Pros

  • Zero perceptual lag — fastest possible response.
  • No motion to lose at 60fps under load — animation stutter is impossible.
  • `prefers-reduced-motion` users see the same behaviour everyone else does.
  • Lowest implementation cost — just `display: block`/`none`.
  • Predictable hover-grace becomes the only timing concern (Q6).

Cons

  • Reads as "system chrome", not as part of the editorial register.
  • The dark-on-cream contrast jolt of opening the C ink-fill panel is at full intensity instantly — visually startling on every hover.
  • No softening of sticky-scroll behaviour — the panel just appears mid-scroll.
  • Misses an opportunity to use the redesign's --motion-fast token (already defined for chrome transitions).

When this is right

  • If feel matters less than feedback latency.
  • If accessibility is the dominant concern (no motion at all).
  • If we want to firmly distance the dropdown from "fanzine artefact" and call it pure UI chrome.
