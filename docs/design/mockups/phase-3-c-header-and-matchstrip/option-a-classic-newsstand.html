PHASE 3 · CHECKPOINT C · DIRECTION A ★ CLASSIC NEWSSTAND 2026-05-03

Een masthead op cream papier.

The header reads like the front page of a daily paper: KCVV ELEWIJT wordmark on the left, horizontal nav in mono caps through the centre, search glyph and an ink-fill WORD LID button hard right. Below it, a single hairline ink rule and a persistent <MatchStrip> band that carries the next fixture, the live score, or the result — and disappears entirely when no upcoming match exists.

★ 01 · SITEHEADER — DESKTOP DEFAULT — wordmark · nav · search · WORD LID

Het masthead.

Three columns. Wordmark anchors the left like a newspaper banner; the nav floats centre in IBM Plex Mono caps; the search glyph and ink-filled WORD LID hold the right. Bottom: a 1px ink rule that doubles as the seam where the MatchStrip begins.

↑ FOR

Reads instantly as a club masthead — wordmark left, content centre, action right is the universal newspaper grammar. Nav in mono caps echoes the same MonoLabel token the rest of the redesign uses, so the header doesn't introduce a new typographic voice. Single hairline ink rule keeps the seam between header and MatchStrip unambiguous.

↓ AGAINST

Centred nav demands real estate — at 1024–1100px the gap between wordmark and nav tightens and the layout starts to feel cramped. Mitigation: nav collapses to hamburger earlier than a typical breakpoint (≤ 960px). Mono-caps nav is also less skimmable than sentence-case for first-time visitors; the active-link underline does most of the wayfinding work.

★ 02 · SITEHEADER — MOBILE — hamburger toggle · collapsed nav · scaled wordmark

Op een telefoon.

↑ FOR

Hamburger left + wordmark centre + actions right is the canonical mobile pattern — zero learning curve. Compact LID abbreviation keeps the primary CTA above the fold without colliding with the wordmark. Hamburger uses the same 1.5px ink stroke as the search glyph and dismiss control elsewhere — no new affordance shape.

↓ AGAINST

Truncating WORD LID to LID sacrifices verb-noun clarity for fit. Alternative: drop the search glyph on mobile and keep the full label — but search is heavily used on news pages. The drawer/sheet that opens behind the hamburger is out of scope for this checkpoint and will need its own design pass.

★ 03 · MATCHSTRIP — STATE MATRIX — four states stacked · same chrome, different payload

Vier toestanden.

The strip has one chrome (cream band, ink top + bottom rules, shield · meta · CTA grid) and four payloads: no match (renders nothing), upcoming, live, and concluded. The shield is always KCVV-left, opponent-right; competition / date / venue or score sit in the centre; CTA holds the right edge.

↑ FOR

One chrome, four payloads — the strip's structural grid (shield · meta · CTA) never changes between states, so the user's eye lands on the same coordinates whether the next match is three days out or already 67 minutes in. Score uses Playfair italic 900, echoing the wordmark and big editorial headlines — no new typography. LIVE pill is the single moment the alert-red token enters the chrome and earns its salience.

↓ AGAINST

Four states means four content-shape variations the editorial team must keep populated correctly through automation — a stale "live" state that doesn't transition to "concluded" looks broken. The state 1 placeholder shown here is a documentation artefact only; in production the component renders null and there is no visible seam left behind. The minute marker uses a Unicode prime (67ʼ) for the prototype; production should switch to min abbreviation if the prime renders inconsistently across fonts.

★ 05 · PRIMITIVE REUSE MAP — each chrome part traced back to its source primitive

Wat hergebruikt wat.

Every visible part of the header and strip composes onto an existing locked primitive. No bespoke chrome — every box, label, and CTA inherits a vocabulary already shipped or planned.

↑ FOR

Every line in the table maps to a primitive that's already specified or shipped — Button, MonoLabel, EditorialHeading, ClippedCard, StampBadge, TicketStub, IconButton. No new primitives required for either component. The header + strip is a composition exercise, not an invention exercise.

↓ AGAINST

The shield is the one concession — it borrows StampBadge's geometry but introduces a new heraldic clip path. If shield variants multiply (B-team, jeugd, away kit colours), it may earn its own ShieldFigure primitive next phase. Score typography also doubles down on Playfair italic 900 — used in the wordmark, headings, and now the score, this is a deliberate concentration that risks a single-voice fatigue if the strip is on every page.

★ NOTES — resolved decisions, open questions, token map

Resolved

  • Header + MatchStrip behave as a single sticky unit — visually unified by hairline ink rules, not shadows
  • Wordmark uses Playfair italic 900 with jersey-deep accent on Elewijt, no logomark badge
  • Nav links: IBM Plex Mono caps, 12px, weight 600, tracking 0.04em — slightly tighter than the standard MonoLabel 0.08em to fit six items at 1024px
  • Active nav link: jersey-deep colour + 2px jersey-deep underline via ::after
  • WORD LID is the canonical ink-fill primary button: shadow-paper-sm default, press-down on hover
  • Search uses the typographic glyph ⌕ in the prototype; production swaps in Phosphor MagnifyingGlass
  • Mobile breakpoint ≤ 960px → hamburger replaces nav, wordmark drops sup, WORD LID truncates to LID
  • MatchStrip is a single-row grid (auto · 1fr · auto) — fixture left, meta centre, CTA right
  • State 1 (no match) → component returns null; the hatched placeholder shown here is documentation-only
  • State 3 (live) is the only state that introduces alert-red into the chrome, via the LIVE pill
  • Score uses Playfair italic 900 at 28px — same voice as the wordmark, scaled for inline data display
  • No radial glows, no rim-lights, no gradient surfaces — paper-flat throughout
  • No tape, no rotation on either component — the masthead reads as official chrome, not editorial scrap

Open

Owner pick between this Direction A (Classic Newsstand) and any siblings authored alongside. Open question for the lock-in: should the strip carry a fourth secondary state — postponed / cancelled — or should those use the page-level <Alert variant="warning"> primitive from Phase 2 instead? Recommendation: defer to <Alert>; the strip stays focused on the four states shown here.
