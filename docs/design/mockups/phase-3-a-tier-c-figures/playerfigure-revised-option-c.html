Direction C — Matchday Programme · PlayerFigure (revised) · Phase 3 · Checkpoint A

Corrected after the master-design hybrid (photo face stitched into a drawn jersey/arms) was rejected for reading as a Mickey-Mouse pastiche. PlayerFigure now ships as two mutually-exclusive states: a quiet rectangular photo inside a 1px ink rule frame, or a full-figure 1px line-art drawing on cream paper — never a mash-up of both. The sans-mono name plate beneath is identical across states.

★ Rejected direction The original master-design PlayerFigure composited a photographic face on top of a drawn jersey + drawn arms. That hybrid reads as a costume ("Mickey Mouse") and is no longer the spec. Photo and illustration are now disjoint states of the same component.

★ STATE A PHOTO ONLY

Eén beeld, één regel.

The photo carries the figure entirely on its own. No drawn shapes overlap the image — no jersey contour, no arms, no stylised head. The frame is a thin 1px ink rule with an 8px ink paper shadow; the shadow gives the figure its weight, the rule gives it its discipline.

  • Source: rectangular psdImage (3:4 aspect, full bleed inside frame)
  • Frame: border: 1px solid ink + box-shadow: 8px 8px 0 0 ink
  • Inner inset: 10px hairline rule in cream-on-image — programme-booklet convention
  • Treatment: filter: contrast(1.05) saturate(0.92) — printed-paper feel, no rim light, no glow
  • Corner stamp: mono caps PSD-2026 top-left, ink-bordered cream box
  • Plate: sans-mono caps, 11px, #NN in jersey-deep, name in ink, role in ink-muted
  • Surrounding TapedCard carries the paper feel; the figure stays disciplined inside

↑ FOR

Reads instantly as a matchday-programme entry — quiet, editorial, deferential to the photographer. The rectangular photo respects the actual psdImage aspect (~90% of players ship rectangular sources, per the subject_photo_fallback memo) so no re-cropping is needed at the data layer. The plate is the only colour spike (#NN in jersey-deep), keeping the figure monochrome-with-a-flash. No drawn shapes anywhere — eliminates the rejected costume reading.

↓ AGAINST

Quieter than tier-A or tier-B figures — won't carry a hero on its own without a TapedCard rotation + tape strip surrounding it. The hairline inner inset rule is faint by design (cream at 60% opacity over the photo) and may disappear on very dark image regions; acceptable trade-off for keeping the figure paper-thin. Image filter is conservative — owners who want a stronger duotone treatment will need a follow-up token.

★ 02 · ILLUSTRATION STATE — full 1px line-art figure · jersey + arms + stylised head · NO photo anywhere

De tekening.

Used when there is no psdImage, or when the editorial context calls for a drawn figure (youth scouting reports, a tribute column, a transfer rumour). The drawing is a full figure — jersey, both arms, stylised oval head — rendered in 1px stroke only. No fills, no shading, no facial features. Same frame, same inner inset rule, same name plate beneath as the photo state.

The illustration fallback is canonical — identical across all photo-state directions. Direction A/B/C governs only the photo state (above).

CANONICAL illustration fallback — shared across directions A · B · C
#8 Maxim Breugelmans · A-PLOEG

↑ FOR

Solves the photo-missing case without falling back to a generic placeholder rectangle. The drawing is recognisably a player figure (jersey, arms, head) but unmistakably a drawing — no chance of mistaking it for a real photo, no chance of reading as a costume since there is no photographic face. Same frame and same plate as the photo state means a roster grid mixing photo'd and un-photo'd players reads as a single editorial artefact.

↓ AGAINST

Ships a small SVG library — over time the youth/staff/keeper variants will grow (different jersey cuts, different number sizes). Drawing one figure is fine; six is a quiet maintenance tax. The featureless oval head reads as deliberate-and-editorial inside a programme spread; outside that frame (e.g. on a roster grid alongside many photo'd players) it can read as a placeholder. Acceptable: the placement is always editorially chosen, never automatic.

Dezelfde foto, dichterbij.

A scaled-up CSS crop of the same source — no second image, no separate variant. The face zone fills roughly two-thirds of the frame, leaving the corner stamp and the inset rule untouched. This is an art-direction switch, not a different component.

  • Source: same rectangular psdImage as the default photo state
  • Crop: object-position: 50% 18% + transform: scale(1.35) + transform-origin: 50% 25%
  • Frame: identical 1px ink rule + 8px paper shadow
  • Plate: identical sans-mono row — chest-up crop never changes the metadata
  • Corner stamp: PSD-2026 · CROP annotates that the source was art-directed
  • Use cases: InterviewHero, PlayerHero, long-form lead, transfer announcement

↑ FOR

Same component, same source, same plate — only the crop changes. No fork in the figure tree, no second image asset. The CSS crop is non-destructive: switching isTight off restores the default 3:4 framing instantly. Editors get art-direction headroom without a Sanity-side asset duplication.

↓ AGAINST

Auto-cropping a wide-angle psdImage to a tight chest-up framing produces uglier edges on some sources (player off-centre, helmet partly cut, etc). Mitigation: isTight is editor-opt-in per article — never automatic. The fixed scale value (1.35) is a compromise between "head fills frame" and "shoulders still visible"; very tall portraits may need a per-document override token in a follow-up.

↑ FOR

Every chrome decision in this revised PlayerFigure maps to a primitive already shipped on main. Migration risk is the lowest of the three Tier C directions — the only "new" material is the inner inset rule (a single CSS pseudo-element) and the line-art SVG (a single asset). Photo and illustration share the frame and the plate, so the component contract stays small: one prop selects the state, one prop opts into the tight crop.

↓ AGAINST

Two states means the consumer has to know which to render — the host page (PlayerHero, RosterGrid, etc.) needs a small piece of conditional logic that asks "is there a psdImage?" before picking the state. Acceptable: that decision lives in the data layer (player.psdImage is nullable in the Sanity schema) and is a one-line check at the call site.

★ NOTES — direction-level summary · structure decisions · production handoff

Resolved

  • PlayerFigure ships as two mutually-exclusive states: photo OR illustration. Never both.
  • Photo state uses a rectangular psdImage inside a 1px ink rule frame with 8px paper shadow.
  • Illustration state is a full-figure 1px line-art drawing — jersey + arms + stylised oval head — on cream paper. No fills, no shading, no facial features.
  • Both states share the same frame, the same inner inset rule, and the same sans-mono name plate beneath.
  • Name plate format: #NN · NAME · POSITION — #NN in jersey-deep is the only colour spike.
  • Cropped-tight variant of the photo state uses object-position + transform: scale on the same source — no second image asset.
  • Surrounding TapedCard has no rotation; TapeStrip and TicketStub are optional ornaments on the host card, not the figure.
  • Press-down hover canon applied per memo: hover:translate-x-1 hover:translate-y-1 hover:shadow-none + transition-all duration-300.
  • No emoji, no Lucide — typographic glyphs only. Stenciletta off-limits per visual-preferences memo.
  • No radial glows, no rim-lights — chrome stays paper-thin throughout.

Rejected

The original master-design hybrid (photographic face composited onto a drawn jersey + drawn arms) is rejected for reading as a costume ("Mickey Mouse"). The corrected spec disjoins the two: photo OR illustration, never composited. This file is the canonical reference for that correction.

Open

Owner pick on the illustration's stroke weight (1.25px here vs 1px in the JerseyShirt mockup) — needs alignment across PlayerFigure + JerseyShirt before shipping. SVG library size (one figure now, six variants planned) needs a directory plan inside apps/web/src/components/design-system/PlayerFigure/illustrations/.
